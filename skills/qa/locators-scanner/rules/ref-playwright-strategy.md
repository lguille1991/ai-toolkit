## Philosophy

Think like a user, not like a machine. Prioritize user-facing attributes that reflect how people and assistive technologies perceive the page.

## Locator Hierarchy (first match wins, top to bottom)

| Priority | Strategy | When to use | Code pattern |
|---|---|---|---|
| 1 | `getByRole` | Element has an ARIA role (explicit or implicit) with a distinguishable accessible name | `page.getByRole('button', { name: 'Submit' })` |
| 2 | `getByLabel` | Form control with an associated `<label>` element | `page.getByLabel('Email Address')` |
| 3 | `getByPlaceholder` | Input/textarea with `placeholder` attribute (no label available) | `page.getByPlaceholder('Enter your email')` |
| 4 | `getByText` | Non-interactive element identifiable by unique visible text | `page.getByText('Welcome back')` |
| 5 | `getByTestId` | Element has `data-testid` attribute | `page.getByTestId('login-form')` |
| 6 | `getByAltText` | Image with `alt` attribute | `page.getByAltText('Company logo')` |
| 7 | `getByTitle` | Element with `title` attribute | `page.getByTitle('Close dialog')` |
| 8 | CSS selector | No semantic attribute available; use minimal CSS | `page.locator('#unique-id')` or `page.locator('[data-cy="submit"]')` |
| 9 | XPath | **Last resort** — only when CSS cannot express the query | `page.locator('xpath=//div[@class="wrapper"]/button')` |

## Framework-Specific Rules

- Always prefer `getByRole` as the first attempt. Most interactive elements have implicit roles.
- For `getByRole`, use `{ exact: true }` when the accessible name could partially match other elements.
- If `getByRole` matches multiple elements, use `.filter()` or chain with a parent locator.
- Avoid XPath — Playwright docs explicitly call it a bad practice.
- For chaining: `page.getByRole('listitem').filter({ hasText: 'Product 2' }).getByRole('button', { name: 'Add to cart' })`
